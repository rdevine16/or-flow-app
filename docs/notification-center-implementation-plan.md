# Implementation Plan: Notification Center Revamp

> Generated by pre-implementation interview on 2026-03-02

## Summary

Revamp the notification system from disconnected tables and computed dashboard alerts into a unified notification center. Slide-out panel with two sections (Active Alerts + Persistent Notifications), Supabase Realtime for live updates, deep-link click-through, and a creation pipeline that respects existing facility notification settings. Initial scope: integration events (HL7v2 case auto-created/updated/cancelled) + data quality notifications. 6 phases, ~6-8 sessions.

## Interview Notes (Key Decisions)

| # | Decision | Choice |
|---|----------|--------|
| 1 | Dashboard alerts vs. notifications | Keep both, separate sections in slide-out panel |
| 2 | Retention model | Remove 24hr default. Read: 30 days. Unread: 90 days. |
| 3 | `data_quality_notifications` table | Consolidate into main `notifications` table |
| 4 | Creation pipeline | DB function `create_notification_if_enabled()` + edge functions |
| 5 | UI layout | Slide-out panel (right side), overlay with backdrop |
| 6 | Initial scope | Integration events + Data quality only |
| 7 | Realtime | Supabase Realtime subscription on `notifications` |
| 8 | Click-through | Deep links derived from type + metadata |
| 9 | Schema extension | Add `metadata JSONB` + `category TEXT` to `notifications` |
| 10 | Panel style | Overlay with backdrop (matches case drawer) |
| 11 | HL7v2 notification source | In the HL7v2 edge function directly |
| 12 | Filtering | Tabs: All / Unread |
| 13 | DQ notification source | `run-data-quality-detection` edge function |
| 14 | Alert dismissal | Dismissable until next detection |
| 15 | Cleanup retention | 30 days read, 90 days unread |

## Phase 1: Schema Migration & Notification Pipeline

**Complexity:** Medium

### What it does
- Extend `notifications` table with `category`, `metadata` JSONB, make `expires_at` nullable
- Drop `data_quality_notifications` table (consolidate into `notifications`)
- Create `create_notification_if_enabled()` DB function that checks `facility_notification_settings` before inserting
- Create `clean_old_notifications()` DB function for retention cleanup
- Add indexes for panel queries
- Add new notification types to `notification_settings_template` for integration events

### Files touched

| File | Change |
|------|--------|
| `supabase/migrations/YYYYMMDD_notification_center_schema.sql` | **Create** — ALTER `notifications` (add category, metadata, make expires_at nullable), DROP `data_quality_notifications`, CREATE FUNCTION `create_notification_if_enabled()`, CREATE FUNCTION `clean_old_notifications()`, add indexes |
| `lib/dal/notifications.ts` | **Create** — DAL functions: `getNotifications()`, `getUnreadCount()`, `markAsRead()`, `markAllAsRead()`, `createNotification()` (for app-level creation) |

### Key implementation notes
- `create_notification_if_enabled()` takes: `p_facility_id`, `p_type`, `p_title`, `p_message`, `p_category`, `p_metadata`, `p_case_id`, `p_sent_by`
- It queries `facility_notification_settings` for the given `facility_id + notification_type`
- If `is_enabled = false` OR `'in_app'` not in `channels` array → return NULL (no notification created)
- If enabled → INSERT into `notifications` with NULL `expires_at` (persistent)
- Special case: `patient_call` type keeps its existing 24hr expiry behavior (don't break CallNextPatientModal)
- `clean_old_notifications()` deletes: notifications that have a `notification_reads` entry where `read_at < now() - 30 days` AND notifications with no read entry where `created_at < now() - 90 days`. Excludes `patient_call` type (handled by existing `clean_expired_notifications()`).
- Add `case_auto_created`, `case_auto_updated`, `case_auto_cancelled`, `data_quality_issue` to `notification_settings_template` if not already present. Seed `facility_notification_settings` for existing facilities.
- DAL functions use `useSupabaseQuery` pattern. `getNotifications()` supports pagination (offset/limit) and filter (all/unread via LEFT JOIN to `notification_reads`).
- `markAsRead()` inserts into `notification_reads` (notification_id, user_id). `markAllAsRead()` bulk inserts for all unread notifications.

### Commit message
`feat(notifications): phase 1 - schema migration and notification creation pipeline`

### 3-stage test gate
1. **Unit:** Test `create_notification_if_enabled()` — creates notification when type is enabled, returns NULL when disabled. Test `clean_old_notifications()` — removes old read/unread correctly, preserves patient_call. Test DAL functions (getNotifications, markAsRead, getUnreadCount) with mock data.
2. **Integration:** Create a facility with notification settings → call `create_notification_if_enabled()` for enabled type → verify notification appears in `getNotifications()`. Call for disabled type → verify no notification created. Mark as read → verify `getUnreadCount()` decreases.
3. **Workflow:** Enable integration type in facility settings → create notification → fetch in DAL → mark as read → verify read state → run cleanup → verify old notifications removed.

## Phase 2: Notification Panel UI

**Complexity:** High

### What it does
- Create the slide-out notification panel component (replaces current dropdown)
- Two sections: Active Alerts (from `useDashboardAlerts()`) and Persistent Notifications (from DB)
- All / Unread tabs for the notifications section
- Individual notification cards with unread dot, icon, title, message preview, timestamp
- Mark as read / Mark all read actions
- Deep-link click-through navigation
- Empty states for both sections
- Dismiss button on active alerts (dismissable until next detection)

### Files touched

| File | Change |
|------|--------|
| `components/global/NotificationPanel.tsx` | **Create** — Slide-out panel with backdrop overlay, two sections, tabs, scroll area |
| `components/global/NotificationCard.tsx` | **Create** — Individual notification card (unread dot, icon, title, message, timestamp, click-through) |
| `components/global/ActiveAlertCard.tsx` | **Create** — Alert card with dismiss button, priority dot, icon, click-through |
| `components/global/NotificationBell.tsx` | **Modify** — Replace dropdown with panel trigger. Badge = unread notifications + active alerts. |
| `lib/hooks/useNotifications.ts` | **Create** — Hook wrapping `getNotifications()` DAL with pagination state and Realtime subscription |
| `lib/hooks/useUnreadCount.ts` | **Create** — Hook for unread badge count, Realtime subscription for live updates |
| `lib/hooks/useDashboardAlerts.ts` | **Modify** — Add dismissed state (sessionStorage-backed, resets on page refresh or data refetch) |

### Key implementation notes
- Panel uses `fixed inset-y-0 right-0 w-96 z-50` with `transition-transform` for slide animation
- Backdrop: `fixed inset-0 bg-black/30 z-40` with click-to-close
- Active Alerts section is collapsible with a header showing count badge
- Dismiss state stored in `sessionStorage` keyed by alert ID. On `useDashboardAlerts` refetch, dismissed IDs are cleared if the alert no longer exists, kept if it still does.
- Notification cards: click navigates to `metadata.link_to` and marks as read. Hover shows "Mark as read" button.
- Pagination: initial fetch of 20 notifications, "Load more" button at bottom (or intersection observer for infinite scroll)
- `useNotifications()` subscribes to Realtime channel `notifications:facility_id=eq.{facilityId}` for INSERT events. On new notification, prepend to list and increment unread count.
- `useUnreadCount()` is a lightweight separate hook — only fetches count (not full notifications) for the bell badge. Also subscribes to Realtime for live badge updates.
- Tab state (All/Unread) is local component state. Switching tabs refetches with appropriate filter.
- "Mark all read" calls `markAllAsRead()` DAL function → optimistic UI update → refetch unread count.
- Notification type → icon mapping config (similar to existing ALERT_ICONS pattern)
- Notification type → route mapping for deep links: `case_auto_created` → `/cases/{case_id}`, `data_quality_issue` → `/data-quality`, etc.

### Commit message
`feat(notifications): phase 2 - notification panel UI with realtime and deep links`

### 3-stage test gate
1. **Unit:** NotificationPanel renders with empty state. NotificationCard displays title, message, timestamp, unread dot. ActiveAlertCard dismiss button works. Bell badge shows correct count. Tab switching filters notifications.
2. **Integration:** Open panel → shows active alerts from `useDashboardAlerts()` + notifications from DB. Mark notification as read → unread dot disappears, count decrements. Mark all read → all dots clear. Dismiss alert → alert disappears from panel.
3. **Workflow:** Bell shows badge (3) → click bell → panel slides open → Active Alerts section shows 1 alert → Notifications section shows 2 unread → click notification → navigates to case detail → return → notification now read → click "Mark all read" → badge clears.

## Phase 3: Integration Event Notifications (HL7v2)

**Complexity:** Medium

### What it does
- Add notification creation in the HL7v2 edge function after successful case processing
- Create notifications for: case auto-created (S12), case auto-updated (S13/S14), case auto-cancelled (S15/S16)
- Respect facility notification settings (only create if type is enabled and `in_app` is in channels)
- Store relevant metadata (case_id, patient name, procedure, link_to URL)

### Files touched

| File | Change |
|------|--------|
| `supabase/functions/hl7v2-listener/index.ts` | **Modify** — After case creation/update/cancel, call `create_notification_if_enabled()` via RPC |
| `supabase/functions/hl7v2-listener/notification-helper.ts` | **Create** — Helper to build notification title/message/metadata from SIU parsed data |

### Key implementation notes
- After the import service processes a message and creates/updates/cancels a case, call the `create_notification_if_enabled()` DB function via Supabase RPC
- For S12 (case created): `type='case_auto_created'`, title includes patient name + procedure, metadata includes `{ link_to: '/cases/{case_id}', case_id, patient_name, procedure_name, source: 'epic_hl7v2' }`
- For S13/S14 (case updated): `type='case_auto_updated'`, title includes patient name + summary of changes, metadata includes `{ link_to: '/cases/{case_id}', changes: ['rescheduled', 'room changed', etc.] }`
- For S15/S16 (case cancelled): `type='case_auto_cancelled'`, title includes patient name, metadata includes `{ link_to: '/cases/{case_id}', reason }`
- `sent_by` is NULL for integration-sourced notifications (no human sender)
- `category` is `'Case Alerts'` for all integration types
- Do NOT create notifications for messages in `pending_review` status — only after successful case processing
- Do NOT create notifications when running from the test harness (check for a flag or test mode indicator)

### Commit message
`feat(notifications): phase 3 - integration event notifications from HL7v2 edge function`

### 3-stage test gate
1. **Unit:** `buildNotificationFromSIU()` helper produces correct title/message/metadata for each trigger event (S12, S13, S14, S15, S16). Returns null for pending_review messages.
2. **Integration:** Send test SIU^S12 via test harness → case created → notification appears in `notifications` table with correct type, title, metadata, link_to. Disable `case_auto_created` in facility settings → send another S12 → no notification created.
3. **Workflow:** Send S12 via test harness → open notification panel → new notification appears (via Realtime) → click notification → navigates to case detail page → case data matches.

## Phase 4: Data Quality Notifications

**Complexity:** Medium

### What it does
- Add notification creation in the `run-data-quality-detection` edge function
- When new issues are detected, create a summary notification
- One notification per detection run (not per issue — avoids flooding)
- Respect facility notification settings

### Files touched

| File | Change |
|------|--------|
| `supabase/functions/run-data-quality-detection/index.ts` | **Modify** — After detection run, if new issues found, call `create_notification_if_enabled()` |

### Key implementation notes
- After the detection function finishes and has found N new issues, create one notification: `type='data_quality_issue'`, title: "Data Quality: {N} new issues detected", message: brief summary of issue types found
- `metadata`: `{ link_to: '/data-quality', issues_count: N, issue_types: ['validation', 'missing_data', ...] }`
- `category`: `'Reports & Summaries'` (matches the existing settings category)
- Only create if `issues_count > 0` (no notification for clean runs)
- Dedup: if a `data_quality_issue` notification was created in the last hour for the same facility, skip (prevents flooding from frequent detection runs)
- `sent_by` is NULL (system-generated)

### Commit message
`feat(notifications): phase 4 - data quality detection notifications`

### 3-stage test gate
1. **Unit:** Notification creation logic produces correct title/metadata for various issue counts. Dedup logic skips when recent notification exists.
2. **Integration:** Run data quality detection on a facility with known issues → notification created in DB with correct metadata. Run again within 1 hour → no duplicate notification. Disable `data_quality_issue` in facility settings → run detection → no notification.
3. **Workflow:** Trigger data quality detection → notification appears in panel (Realtime) → click → navigates to /data-quality page → issues visible.

## Phase 5: Active Alert Dismissal & Polish

**Complexity:** Low-Medium

### What it does
- Implement dismissable active alerts with session-scoped memory
- Polish notification panel animations, loading states, error states
- Handle edge cases: notification deleted while panel open, facility switch, auth state changes
- "View all on Dashboard" link in Active Alerts section footer

### Files touched

| File | Change |
|------|--------|
| `lib/hooks/useDashboardAlerts.ts` | **Modify** — Add `dismissAlert(id)` and `isDismissed(id)` with sessionStorage backing |
| `components/global/NotificationPanel.tsx` | **Modify** — Wire dismiss buttons, add loading/error states, animation polish |
| `components/global/ActiveAlertCard.tsx` | **Modify** — Add dismiss button handler, fade-out animation |
| `components/global/NotificationBell.tsx` | **Modify** — Handle facility switch (reset subscriptions), auth state changes |

### Key implementation notes
- `sessionStorage` key: `dismissed_alerts` → JSON array of alert IDs
- On `useDashboardAlerts` data refetch: check each dismissed ID — if the alert still exists in new data, keep dismissed. If it no longer exists (issue resolved), remove from dismissed list.
- This means: dismiss "5 cases flagged" → fix 2 cases → next refresh shows "3 cases flagged" (alert returns because condition changed)
- Panel slide animation: `transform: translateX(100%)` → `translateX(0)` with `transition-transform duration-300 ease-out`
- Loading state: skeleton cards while fetching notifications
- Error state: "Failed to load notifications" with retry button
- Keyboard: Escape closes panel
- On facility switch (effectiveFacilityId changes): close panel, reset notification list, re-subscribe Realtime
- Mobile responsive: panel takes full width on small screens

### Commit message
`feat(notifications): phase 5 - alert dismissal and panel polish`

### 3-stage test gate
1. **Unit:** Dismiss alert → disappears from rendered list. Refresh data → dismissed alert returns if condition persists. SessionStorage write/read works correctly.
2. **Integration:** Dismiss alert → panel re-renders without it → refresh page → alert shows again (session cleared). Switch facility → panel resets, new facility's alerts load.
3. **Workflow:** Open panel → dismiss alert → close panel → re-open → alert still dismissed → refresh page → alert visible again (session cleared) → fix underlying issue → alert gone permanently.

**Complexity:** Low

- Verify the full system end-to-end
- Performance audit (query plans, Realtime subscription count)
- Verify CallNextPatientModal still works unchanged
- Run security advisors for new migration

### Files touched

| File | Change |
|------|--------|
| `supabase/migrations/YYYYMMDD_notification_cleanup_cron.sql` | **Create** — Schedule `clean_old_notifications()` to run daily via pg_cron (or document manual edge function approach if pg_cron unavailable) |
| Various test files | **Create/Modify** — End-to-end integration tests for the full notification flow |

### Key implementation notes
- If pg_cron is available: `SELECT cron.schedule('clean-old-notifications', '0 3 * * *', 'SELECT clean_old_notifications()');` (runs at 3 AM daily)
- If pg_cron is not available: create a lightweight edge function `clean-notifications` that calls `clean_old_notifications()` and set up an external cron (e.g., Supabase's built-in cron or a GitHub Action)
- Performance checks: ensure `getNotifications()` query uses indexes (EXPLAIN ANALYZE), verify Realtime subscription doesn't leak on unmount
- Verify CallNextPatientModal flow: create patient call → notification appears with 24hr expiry → expires correctly
- Run `get_advisors` for security check on new migration (RLS coverage, missing policies)
- Test with multiple users: User A creates case via HL7v2 → User B sees notification in their panel (same facility)

### Commit message
`feat(notifications): phase 6 - auto-cleanup, performance audit, and final testing`

### 3-stage test gate
1. **Unit:** `clean_old_notifications()` correctly identifies and removes old read (30 day) and unread (90 day) notifications. Preserves patient_call notifications. Preserves recent notifications.
